# Memo: Should the gallery PNGs stay committed in-tree as re-rendered build outputs?

> **Ruling (Mike, 2026-06-10):** keep all 32 committed — including the 22 text
> scenes the memo flags as repo-weight waste. We are following the text-
> determinism arc (task #5 / [text-boundary.md](../text-boundary.md)), whose
> slice D records the gallery's text programs and replays them byte-identically
> on CI — that brings the text scenes *into* the byte-identity gate rather than
> dropping them from the tree, so their weight earns all three jobs (review,
> gate, determinism anchor) like the other 10. The memo's A5 split is therefore
> declined. The one accepted engineering fix is §4.4 — replace the gallery
> output **glob** with a **static list** so the build's description no longer
> depends on its own outputs existing (the `generator = 1` clean-patch becomes
> defense-in-depth).

**Scope read:** `README.md` (the Gallery section's 32 embeds + Quick start's "review and commit the new PNGs alongside the code" lockstep framing), `.github/workflows/gate.yml` (the byte-identity step over 10 text-free scenes), `configure.py` (the `run_gallery` rule + the `images` edge, lines 179–215, 289–404), the `generator = 1` commit `ebc6f5d` ("ninja -t clean no longer deletes the committed gallery PNGs"), `docs/text-boundary.md` (the determinism arc: recorded programs replay byte-identically, committed PNGs as the cross-machine anchor), and the sibling decision records `codec-outsourcing.md`, `metal-backend.md`, `security-review.md` for genre and voice.

**Measured (this machine, this repo):**
- Working tree: **32 PNGs, 1.6 MB** (`du -sh gallery`); as committed blobs, **1.12 MB on-disk** in the pack, 1.20 MB raw.
- History: **256 distinct PNG blobs across all of git, 7.65 MB on-disk** — **~26% of the 29 MB `.git`** — against 42.46 MB if those blobs were stored inflated.
- **PNGs are stored whole: zero deltas.** Every current PNG has a null delta-base; `emojiscale.png` packs 265,297 → 251,612 bytes (5%). Git's delta+zlib does essentially nothing to already-deflated PNGs, so **history cost is purely additive** — each distinct revision is a fresh whole blob, forever.
- **57 of 299 commits touch `gallery/`** (~19%); historical mean **~137 KB on-disk added per gallery-touching commit**. Churn is concentrated: `text.png` has **21 revisions**, `gradients.png` 16, `shapes.png` 13.
- **The gate byte-checks 10 of the 32 scenes** (text-free only); the other 22 text-bearing scenes re-render on every `ninja` but are *not* byte-gated (host font dependence — see `gate.yml`'s comment about Libian TC being download-on-demand).

**One framing correction before anything else.** The tasking says CI byte-diffs "the 10 text-free scenes" — correct — but the decision under attack is broader than the gate: the README commits and embeds **32** PNGs and re-renders all of them every `ninja`. So three distinct things ride on "committed PNGs" and they have *different* strengths: (1) the **review signal** (all 32, inline image diffs), (2) the **CI codegen gate** (10, byte-identity), and (3) the **determinism anchor** (the 10 text-free scenes are the only ones that *can* be a cross-machine byte anchor; the 22 text scenes are host-font-dependent and explicitly excluded). An honest evaluation has to treat these separately, because the alternatives trade against them unequally.

---

## 1. The steelmanned case FOR changing

Taken seriously:

**a. The repo-weight cost is real, additive, and unbounded.** This is the strongest objection and it's measured, not hypothetical. PNGs don't delta — the 256 distinct blobs already committed cost **7.65 MB on-disk, a quarter of the entire `.git`**, and every future rendering change adds a fresh whole copy of each moved scene. A clone pays for all of it forever; git has no mechanism to reclaim it short of history rewriting. Project forward at the historical mean (~137 KB/commit): **+100 gallery commits ≈ +13.7 MB, +500 ≈ +68 MB.** A golden-hash scheme commits ~64 bytes per scene per change — the same 500-commit future costs **kilobytes, not tens of megabytes**. For a learning project whose entire source tree is a few hundred KB of C, a `.git` dominated by binary render outputs is a genuine smell, and it grows monotonically with exactly the activity the project most wants (rendering changes).

**b. The `generator = 1` patch (`ebc6f5d`) is treating a symptom.** The footgun is diagnostic. `run_gallery`'s *outputs* are discovered by globbing `gallery/*.png`; `ninja -t clean` deleted those outputs (correctly — they're build outputs); then the next `configure.py` glob came back empty and emitted `build : run_gallery …` with no outputs, which ninja refuses to load. The fix marks the rule a generator so `clean` spares the PNGs "the same way build.ninja is." But step back: **a build system deleting its own declared outputs broke configuration** because those outputs are *also* tracked source the configurator reads. That circularity — outputs that are inputs to the build's own description — is the defining pathology of committing build artifacts. `generator = 1` papers over the one symptom that surfaced (`clean` + reconfigure); the underlying category confusion (is `gallery/*.png` a source or a product? the build treats it as both) remains, and will surface again at the next tool that reasons about outputs vs. sources (a fresh checkout that hasn't run `configure`, a `clean` build in CI, a `.gitignore` audit). The commit message itself concedes the failure mode is structural by adding a *second* guard (empty-glob → hard error with `git restore gallery/`), i.e. the fix needs its own fix.

**c. Two of the three jobs don't actually need the *bytes*.** The CI codegen gate only needs to detect "did a byte change vs. the blessed render" — that is exactly what a committed SHA-256 per scene delivers, at 64 bytes instead of 30 KB. The determinism anchor (text-boundary.md) needs *reproducibility* — "identical inputs → identical bytes across machines" — which a hash pins as tightly as the bytes do (a hash collision on a real render is not the threat model). Only the **review signal** genuinely needs the bytes, and only for the human, not for any automated check.

**d. The 22 text scenes are dead weight under the current scheme.** They re-render on every `ninja`, churn the working tree, and inflate clones — but they are *explicitly excluded from the gate* because they depend on the host font library. So for 22 of 32 scenes the project pays the full repo-weight and review-churn cost while getting **none** of the CI-determinism benefit. They are committed purely for the inline-diff review value, which is a much weaker justification for permanent binary history than "they anchor reproducibility" (they don't).

## 2. The honest case for the status quo

**a. The inline image diff is a first-class review tool for a graphics library, and only committed bytes deliver it.** This is the decisive point in favor, and it's specific to the domain. When a reviewer opens a PR that changes the rasterizer, GitHub renders **before/after/swipe** image diffs inline from the committed PNGs. For a coverage-AA change, a stroke-join fix, or a gradient-ramp tweak, *seeing the pixels move* is the review — a reviewer can catch a regression (a fringe, a popped mip level, an off-by-one in a join) that no green checkmark would surface. A golden-hash file reduces this to "32 hex strings changed"; the reviewer learns *that* something moved but not *what*, and must check out the branch and render locally to see it. The README's "review and commit the new PNGs alongside the code" is not incidental ceremony — it's the project deliberately buying visual review, and the gallery is also the project's **showcase** (the README embeds all 32 as documentation; those embeds need the bytes in-tree regardless of the gate).

**b. The byte-identity gate is the strongest CI signal available and it's nearly free to keep.** `gate.yml` pins `DEVELOPER_DIR` to Xcode 26.5 specifically so "the trailing git-diff step fails if CI's codegen draws even one different pixel." That's a tolerance-0 oracle over real end-to-end pipeline output (geometry → coverage → blend → PNG encode), catching codegen-level nondeterminism that unit tests with hand-derived constants would miss. The `metal-backend.md` memo leaned on exactly this class of differential as "the project's strongest oracle." A hash gate keeps the *signal* but, per §1c, the bytes are needed anyway for review and showcase — so the gate costs nothing extra on top of bytes already present.

**c. Committed bytes are the literal cross-machine determinism anchor.** `docs/text-boundary.md`'s whole arc is that recorded programs replay byte-identically and the committed PNGs are part of that reproducibility story. The 10 text-free scenes are a concrete artifact saying "this exact pixel output is reproducible from this source on the pinned toolchain." A hash anchors the same claim mathematically, but the *bytes* let a human or a fresh machine actually diff the rendered output, not just compare digests — valuable when diagnosing *why* a determinism gate went red (you can open the new PNG and look).

**d. The cost, in absolute terms, is small and the project is early.** 7.65 MB of PNG history in a 29 MB `.git` is real but not painful at today's scale; a fresh clone is seconds. The repo-weight argument is a *trajectory* argument, and trajectories can be addressed later (a single `git gc`-class history rewrite, or switching schemes prospectively) without having mis-served any reviewer in the meantime. Optimizing clone weight now trades a concrete daily benefit (visual review) against a cost that is currently theoretical.

**e. Generated-on-demand would break the README.** The showcase embeds (`![shapes](gallery/shapes.png)` ×32) render on GitHub *because the bytes are in the repo*. Gitignoring them entirely (the most aggressive alternative) breaks the front-page documentation, which is arguably the project's primary audience-facing surface. Any scheme that removes the bytes has to separately solve "what does the README embed."

## 4. Recommendation

**Keep the gallery PNGs committed — but tighten the model and fix the footgun properly.** The tradeoffs that drove it, in order:

1. **The inline image diff is a domain-specific review tool that no alternative replicates, and it is the single thing only committed bytes can deliver.** For a from-scratch rasterizer where the whole value proposition is "the pixels are correct," letting a reviewer *see* the pixel change in the PR is worth more than the storage it costs. Every byte-free alternative (A2/A3/A4) sacrifices exactly this. That asymmetry is decisive: the cost of the status quo is megabytes of cold history; the benefit is hot, on every graphics PR.

2. **Two of the three jobs (CI gate, determinism anchor) are satisfied by the bytes the review job *already requires*, so they cost nothing marginal.** There's no separate argument to make for committing bytes "for CI" — the gate rides for free on bytes present for review. This is why A3 (golden-hash) is the cleanest engineering answer yet still the wrong call: it optimizes away a cost that's shared across three benefits to save one of them.

3. **But the repo-weight trajectory is real and the 22 text scenes are the waste.** They re-render every `ninja`, they're the biggest scenes (the top three by size are all text/emoji), they churn history, and they are *explicitly outside the gate* — so they pay full cost for only the review/showcase benefit. **Adopt A5:** keep committing the 10 text-free scenes (they earn all three jobs); for the text scenes, either commit them as today purely for showcase value *with eyes open about the churn*, or — better — stop re-rendering them into the tree on every `ninja` and regenerate them on demand for the README, since they can't be a determinism anchor anyway. This roughly thirds the per-commit churn without touching the gate or the visual review of the codegen-critical scenes.

4. **Fix the footgun at the category level, not the symptom.** `generator = 1` stops `clean` from deleting the PNGs, but the real defect is that `configure.py` derives a build edge's *output list* by globbing files that are *also* committed sources. The robust fix is to make the output list **static** (an explicit list in `configure.py`, the same list the gate already hard-codes), so the build description never depends on the presence of its own outputs. That removes the empty-glob failure mode entirely and the need for the second guard, and it's the same posture `gate.yml` already takes by enumerating the 10 scenes by name. (The `generator = 1` patch is fine to keep as defense-in-depth; the point is it shouldn't be load-bearing.)

Honest residue conceded to the change side: even with A5, PNG history keeps growing additively and never deltas, so the `.git` will keep accreting binary weight proportional to rendering activity — a hash scheme would be strictly cheaper to store and would have made `gallery/*.png` clean sources instead of build-output/source hybrids, killing the footgun by construction. The recommendation pays that ongoing storage cost deliberately, in exchange for visual review the project's domain specifically rewards.

## 5. What would change my mind

- **The repo-weight cost becomes felt** — clone time, GitHub's per-repo size limits, or a `.git` that dwarfs the source by an order of magnitude (currently 26%; at, say, 60–70% the trajectory argument flips). Then A3/A5-aggressive wins: keep hashes for the gate, regenerate images for the README via a CI step or a release artifact.
- **The footgun recurs in a new form** — if a fresh-checkout build, a CI `clean`, or a future tool again trips over "outputs that are also globbed sources" despite `generator = 1`, that's evidence the committed-build-output model is structurally wrong, not just rough, and the static-output-list fix should escalate to dropping the bytes.
- **GitHub's image-diff value proves illusory in practice** — if review history shows reviewers don't actually use the inline diffs (they read the code and trust the gate), then the one benefit only-bytes deliver isn't being collected, and A3 becomes correct: keep the tolerance-0 hash gate, drop the bytes, regenerate for the README.
- **The text scenes get a host-independent path** — if Libian TC and the emoji captures become reproducible on a fresh runner (e.g. the font is vendored or the captures are fully serialized per the text-boundary.md arc), the 22 text scenes could *join* the gate, at which point committing their bytes earns all three jobs and the A5 split loses its rationale.
- **A determinism contract weakens** — if the project ever accepts that renders are *not* byte-stable across toolchains (the way `metal-backend.md` re-scoped a global bit-for-bit claim to gate-scoped), the byte-identity gate must become a tolerance gate, the committed-bytes anchor loses its tolerance-0 meaning, and the whole scheme should be re-derived around perceptual diffs rather than byte identity.
- **LFS becomes free** — if the project moves to a host where LFS storage and bandwidth are unmetered and the client is already required, A1's "relocate the weight" stops being a net dependency cost and becomes a clean way to keep bytes + inline diffs while unburdening the main pack.
